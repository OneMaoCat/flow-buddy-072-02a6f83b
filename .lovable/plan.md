

## 开发完成后的人工确认环节设计

### 现有流程分析

当前流程：需求提交 → 计划确认 → 开发执行 → (缺失) → 发布

缺失的环节是「开发完成 → 人工验收」。

### 推荐设计：需求级逐条验收 + 整体确认

```text
需求执行状态流转：
  waiting → running → done → review (人工验收) → accepted / rejected

┌─────────────────────────────────────────────────────────┐
│  需求行（开发完成后）                                    │
│  #3 支付模块集成  ██████████ 100%  🔍待验收              │
│  ┌──────────────────────────────────────────────────┐   │
│  │  展开后显示：                                     │   │
│  │  • Agent 完成详情                                 │   │
│  │  • 变更文件列表 (diff 摘要)                       │   │
│  │  • [✅ 通过]  [❌ 打回并备注]                     │   │
│  └──────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│  统计栏新增：  待验收 12  |  已通过 30  |  已打回 2      │
├─────────────────────────────────────────────────────────┤
│  底部操作栏（当所有需求开发完成时出现）                   │
│  ┌────────────────────────────────────────────────────┐ │
│  │  12 个需求待验收 · 30 已通过 · 2 已打回             │ │
│  │  [一键全部通过]   [批量操作▾]   [返回工作区发布]    │ │
│  └────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 实现步骤

**1. 扩展需求状态类型**
- `RequirementStatus` 新增 `"review"` (待验收) 和 `"rejected"` (已打回)
- 需求开发完成 (`done`) 后自动进入 `"review"` 状态
- 筛选栏新增「待验收」和「已打回」Tab

**2. 需求行增加验收操作**
- `review` 状态的需求行显示橙色「待验收」标识
- 展开后底部增加操作区：「通过」按钮和「打回」按钮（打回需填写备注）
- 打回的需求重新进入 `running` 状态，携带打回原因重新执行

**3. 底部验收操作栏**
- 当存在 `review` 状态需求时，底部显示固定操作栏
- 提供「一键全部通过」快捷操作
- 全部通过后显示「返回工作区发布」按钮，跳转回 workspace 并触发发布流程

**4. 统计卡片更新**
- 新增「待验收」和「已打回」计数卡片
- 进度条区分：开发进度 vs 验收进度
